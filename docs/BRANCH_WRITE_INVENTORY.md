**Reviewed:** 24 July 2026

**Issue:** #41 — Migrate release state for strict main-branch protection

## Current conclusion

Strict pull-request-only protection is **not yet safe to enable**.

The repository now has three workflows that can commit directly to public `main`. Two release orchestrators invoke the reusable production writer but contain no direct public-main push. The remaining writes are limited to fallback monitoring, release recovery and guarded production publication.

Seven workflows now provide read-only immutable evidence: canonical validation, release dry runs, repository audits, dashboard projection, Greasy Fork parity, announcement-state verification and stable update-manifest verification.

`.github/branch-write-inventory.json` is the machine-readable authority. Permanent contracts fail closed when a writer, permission, authority or release-state boundary changes.

## Atomic release state

After Greasy Fork verification, private backup and Discord publication, `release-toolkit.yml` now writes the following together in one commit:

- `status/release-dashboard.json`;
- `status/README.md`;
- `status/update-manifest.json`;
- `.github/greasyfork-version.txt`.

The stable manifest is generated by `.github/scripts/build_stable_update_manifest.py`. The builder refuses publication unless the dashboard records successful validation, GitHub Release publication, Greasy Fork verification, private backup, Discord publication, asset health and release readiness. It validates canonical GitHub and Greasy Fork URLs, semantic version parity and the verified SHA-256.

`publish-update-manifest.yml` no longer publishes or pushes. It runs the builder self-tests, checks the committed manifest against the verified release ledger and retains 30-day evidence. The public manifest URL remains unchanged:

`raw.githubusercontent.com/Conroy1988/missionchief-toolkit-assets/main/status/update-manifest.json`

This removes a workflow dispatch, a waited Actions run and a follow-up bot commit from every normal release.

## Other authority boundaries

- GitHub is canonical; automatic Greasy Fork importing is retired.
- Validation candidates are exact commit/ref/hash artifacts.
- Dashboard Markdown is a deterministic read-only projection.
- Primary dashboard and announcement state are atomic.
- Recovery workflows retain guarded reconstruction until the release-state branch migration is complete.

## Production release sequence

1. Canonical validation uploads exact immutable candidate evidence.
2. Automatic release verifies that exact run and rejects stale commits.
3. Release Readiness independently rebuilds and validates distribution.
4. Production publishes stable distribution and root mirrors.
5. GitHub Release is published and Greasy Fork is verified.
6. The release is backed up privately and announced to Discord.
7. Dashboard JSON, rendered Markdown, update manifest and announcement tracker are committed atomically.
8. GitHub Pages is dispatched and awaited.
9. Dashboard, Greasy Fork, announcement and manifest workflows verify only.

The owner command freshly validates current `main` and starts the same readiness and production workflows.

## Target architecture

- **Public `main`:** reviewed source, workflows, tests, policy and documentation.
- **Distribution branch:** stable `dist/` and root mirrors, written by a scoped release App.
- **Release-state branch:** dashboard, manifest, announcement, fallback-monitor and recovery state.
- **Immutable evidence:** validation, parity, projections, bundles, audits and handovers.
